## 3.4 案例研究：Google Compute Engine

　　Google Compute Engine（GCE）使客户能够在其基础设施上安全高效地运行工作负载。GCE的基础设施支持三种计算实例：性能优化型虚拟机（VM），其中虚拟化资源直接映射到底层硬件；成本优化型VM，客户可使用根据资源需求动态映射到底层硬件的自定义机器配置；以及裸金属实例，客户可直接访问底层计算硬件。

　　图3.9展示了GCE架构的概览。主机的硬件包括用于安全（Titan）和网络及存储虚拟化的专用硬件加速器（Titanium），因此即使是裸金属实例也无法直接访问这些硬件。主机上的I/O加速器本身不执行所有I/O功能，而是将部分功能卸载到网络“悬浮板”（专用高性能数据包处理器）和存储代理。与邮件或搜索等单个服务不同，GCE代表了一个运行其他工作负载的平台。下文我们将讨论与WSC设计相关的该平台的三个关键方面。

### 3.4.1 可信启动

　　Google定制设计的Titan [30] 安全微控制器为加密验证主机上的所有固件和软件 [31] 提供硬件信任根。硬件信任根对主机的引导固件进行加密验证。随后，引导固件验证系统软件，包括GCE用于支持虚拟机实例的内核和基于KVM的虚拟机管理程序。最后，系统软件通过二进制授权验证用户空间软件（如系统守护进程）。如果主机的固件或软件偏离预期状态，则无法访问其在集群中运行所需的凭据。此类机器在机器管理基础设施触发自动修复操作之前，无法参与运行客户工作负载或其他Google服务。

　　可信启动架构提供了两个重要优势。首先，它为分层安全架构奠定基础，上层可以依赖下层（代码和配置）的正确状态（图3.10）。如果没有这一点，即使操作系统本身安全性很强，也可能因固件被篡改而失效。其次，它提供了一种快速从安全漏洞中恢复的方法：如果我们怀疑主机或客户机上存在攻击者，则重启可以保证清除攻击者并恢复默认状态。同一功能也可用于在将机器分配给新客户之前进行清理，即使前一个客户拥有裸金属访问权限。

### 3.4.2 分层卸载

　　历史上，虚拟机依赖主机软件将物理网络接口卡（NIC）及其在物理网络上的物理IP地址转换为虚拟NIC及其在虚拟私有局域网上的虚拟IP地址，通过封装出站数据包和解封装入站数据包实现。出于历史原因，此类虚拟局域网通常被称为VPC（Virtual Private Cloud）。同样，使用PCIe或NVMe接口虚拟化物理I/O设备需要主机支持。

　　如今，GCE通过Titanium [32, 33] 支持客户工作负载在虚拟机或裸金属实例中运行，该卸载系统在主机卸载基础上补充了扩展式卸载（运行在主机外部）。这种架构在安全性、性能和灵活性方面优于基于软件的方法。其他云服务采用类似架构但未分层；例如，Amazon的Elastic Block Store（EBS）Nitro加速器将所有块存储功能集成在单张卡中。这种架构具有简单性优势，但难以灵活调整I/O容量，即无需硬件升级即可调整主机或VM的I/O容量。

　　GCE的主机Titanium适配器是一个基础设施处理单元（IPU）（我们将在第6章详细讨论IPU），用于最小化主机CPU的工作量。它为I/O虚拟化提供硬件加速。在网络方面，它加速了网络数据包的封装/解封装过程，并通过规模优化和卸载优化的PSP安全协议 [34] 加密网络流量。在存储方面，它将虚拟NVMe块设备映射到高性能、灵活、可扩展且持久的Hyperdisk卷。

　　除了主机卸载外，其他（扩展式、主机外）卸载使GCE能够超越每台主机实际可行的范围扩展VM服务。Hyperdisk块存储 [35] 不再局限于主机上加速卡的存储虚拟化性能，而是在两个位置执行该工作。基本处理在本地Titanium适配器上进行，但实际持久化存储处理在单独主机上的扩展式卸载层完成，该层通过Google的集群级文件系统Colossus分布I/O。在传统卸载架构中，更高的块存储IOPS需要更多vCPU的VM（或更大的加速卡），无论工作负载是否需要。这种紧密耦合导致资源浪费和成本增加。Hyperdisk将计算实例大小与存储性能解耦，因为GCE可以独立调整实例本身和存储卸载任务的规模。因此，GCE即使使用相对较小的通用型VM也能支持高存储性能。

　　作为另一个横向扩展卸载的示例，GCE 使用 "hoverboards"[36] 来扩展网络虚拟化。Hoverboards 是独立的软件交换机，充当某些流量的默认路由器。通过 Titanium，Google 的 Andromeda 虚拟网络堆栈在 Titanium 适配器上（再次说明，关于 Andromeda 的更多内容请参见第6章和第7章）会将所有无路由的报文发送到 Hoverboard 网关，这些网关拥有所有虚拟网络的转发信息。

### 3.4.3 配置流程

　　GCE 的控制平面允许用户创建和配置虚拟集群，并让 Google 可靠高效地运行它们（图3.11）。例如，客户可以使用 GCE API 存储他们希望为虚拟机配置的表示，包括 CPU 数量、内存大小、Hyperdisk 卷的数量和尺寸、网络防火墙规则等。

![图3.11 GCE 配置流程](../../build/images/chapter_3/chapter_3_fig_11_GCE_provisioning_flow.png)

图3.11 GCE 配置流程

　　在将虚拟机预期状态的更新存储到 Spanner 后，GCE API 服务器会与多个专用控制平面通信以执行用户的意图。首先，GCE 调用 Borg（第4章将进一步讨论）将虚拟机实例高效地打包到主机上，同时满足其他约束条件，如将同一项目的实例分散到不同主机上，并避免存在待处理维护事件的主机。Borg 随后在目标主机上启动一个虚拟机实例控制器。

　　当虚拟机实例控制器在正确主机上运行后，GCE 会向其发送包含虚拟机基本信息（如虚拟机尺寸）和虚拟设备配置信息（包括 Hyperdisk 卷）的虚拟机规范。Hyperdisk 客户端随后配置一组横向扩展的 Hyperdisk 代理来服务虚拟机的卷，并打开相关的 Colossus 文件。

　　GCE 接着调用 Andromeda[36] Fabric Manager，该组件会在新虚拟机主机的 Titanium 控制器上配置虚拟交换机，以设置与其他虚拟机通信的路由。Fabric Manager 还可能主动编程其他主机上的虚拟交换机，使虚拟网络上的其他虚拟机可以直接向新虚拟机发送报文。或者，它可能省略此步骤，此时发送到新虚拟机的报文将通过 Hoverboard 交换机路由。

### 3.4.4 实时迁移

　　实时迁移（在不关闭虚拟机的情况下将其从一个物理主机迁移到另一个物理主机[37]

　　）展示了这些组件如何协同工作。在虚拟机实例底层硬件的计划维护期间，Compute Engine 可能使用实时迁移将运行中的虚拟机迁移到其他主机。在选择虚拟机进行迁移后，GCE 会通知客户即将发生迁移。经过一段延迟后，迁移过程开始，GCE 使用 Borg 在目标主机上预留容量。

　　GCE 随后开始迁移的预拷贝阶段，在源主机向目标主机拷贝内存页的同时跟踪仍运行的虚拟机所写的任何页面。GCE 同时在预拷贝过程中刷新脏的 Hyperdisk 块到存储，并继续缓冲和写入预拷贝期间更新的任何附加块。

　　迁移的下个阶段是短暂的黑屏阶段，此时源主机上的虚拟机会暂停，其 VCPU 状态转移到目标主机，剩余的脏 Hyperdisk 块被刷新，剩余脏内存页的列表发送到目标主机，Andromeda 网络开始编程 Hoverboards 和其他虚拟机的虚拟交换机，将虚拟机的报文导向目标主机。对于大多数虚拟机，这个黑屏期可以控制在零点几秒内。

　　最后，在后拷贝阶段，GCE 恢复虚拟机。一旦虚拟机恢复运行，GCE 会异步拷贝剩余的脏内存页从源主机到目标主机；如果虚拟机在目标主机获取页面前访问了脏页，该访问会导致页面错误，目标主机从源主机获取页面以允许虚拟机继续运行，中断最小。此外，在分布式 Andromeda 系统更新虚拟交换机以引用虚拟机新位置期间，源主机会将接收到的虚拟机报文转发到目标主机。

　　总之，Google Compute Engine（GCE）是一个优化安全工作负载执行（通过其与 Titan 硬件的验证启动过程）和灵活高性能 I/O（通过分层卸载包括 Titanium IPUs 以及存储和网络的专用横向扩展系统）的平台。高效的虚拟机配置流程和无中断的实时迁移为客户工作负载优化了效率。
