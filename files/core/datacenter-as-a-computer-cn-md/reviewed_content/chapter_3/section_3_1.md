## 3.1 WSC工作负载

### 3.1.1 WSC系统栈

　　典型的WSC部署包含从高层工作负载和应用框架到集群级及监控软件，再到开发与平台软件的多层软件栈。运行在仓库规模计算机（WSC）上的应用程序和软件主导了许多系统设计权衡决策。在本章及下一章中，我们将概述大型互联网服务中软件的一些显著特征，以及构建完整计算平台所需的系统软件和工具。典型WSC部署中的不同软件层包括：图3.1总结了这些层级。在本章中，我们将重点讨论图中上层的两个层级。（第4章将讨论图中的其他组件。）

图3.1 Google仓库规模计算机的软件栈

- 应用级软件：实现特定服务的软件。通常将应用级软件进一步划分为在线服务和离线计算，因为它们往往具有不同的需求。在线服务的示例包括Google搜索、Gmail和Google地图。离线计算通常用于大规模数据分析，或作为生成在线服务所用数据的流水线的一部分，例如构建网络索引或处理卫星图像以创建在线服务的地图瓦片。
- 应用框架：简化应用实现的构建模块：前端或图像服务的服务器基础设施、用户会话分析工具、通用应用元素的UI组件等。
- 集群基础设施：管理资源并提供集群级服务的分布式系统软件集合。最终，我们将这些服务视为数据中心的操作系统。示例包括分布式文件系统、调度器和远程过程调用（RPC）库，以及简化数据中心规模资源使用的编程模型，如MapReduce [1]

　　、Hadoop [2]

　　、Sawzall [3]

　　、BigTable [4]

　　、Dynamo [5]

　　、Dremel [6]

　　、Spanner [7]

　　和Chubby [8]

　　。
- 资源管理：管理与分配资源的基础设施，包括将计算任务调度到特定机器上以及执行存储配额（例如Google的Borg调度器 [9, 10]）。
- 平台软件：所有单个服务器平台中预期存在的通用固件、内核、操作系统发行版和库，以提供基本的机器抽象层。
- 硬件：服务器、加速器（机器学习、视频）、存储设备、网络交换机、数据中心基础设施等。
- 工具：监控应用性能、识别系统瓶颈并测量集群硬件健康状况的软件。

　　由于本书采用WSC的“内部视角”，我们对上述层级的定义不同于另一种常用的行业分类法（如下所述），后者区分基础设施、平台和应用 [11]

　　：

- 基础设施即服务（IaaS）提供计算、存储和网络资源，允许用户运行任意软件。用户不管理或控制底层云基础设施，但可以控制操作系统、存储和部署的应用程序；可能对某些网络组件（如主机防火墙）有有限控制。例如，GCP的IaaS产品GCE [12]

　　提供多种虚拟机和块存储选项。
- 平台即服务（PaaS）提供工具以配置、实例化、运行和管理应用，使开发者和运维人员无需自行构建工具链。例如，GKE [13]

　　和Cloud Run [14]

　　自动化Kubernetes集群管理，Cloud SQL [15]

　　提供完全托管的数据库，Terraform [16]

　　等工具允许标准化配置。
- 软件即服务（SaaS）提供完全托管的应用程序，使客户无需承担与该应用相关的大部分或全部运维任务（例如安装、配置、升级等）。互联网消费者服务始终是SaaS服务，因此该术语主要用于描述企业软件。例如，Google Workspace [17]

　　是SaaS应用。

### 3.1.2 计算消耗模型

　　WSC中的工作负载可以在共享（多租户）或专用硬件配置中运行。它们可以是特定应用（Gmail、YouTube、搜索等），也可以是更广泛的应用框架或平台（GCE、Spanner、BigTable、VertexAI等）。此外，它们可以是内部传统超大规模工作负载，或服务于外部云客户的负载。在Cloud中，客户工作负载可在基础设施即服务（IaaS）平台如Google Compute Engine（GCE）或Google Kubernetes Engine（GKE）上运行，或服务于更高层次的托管服务（例如VertexAI、DataFlow、DataProc、MemoryStore、CloudSQL等）。此外，加速器平台如TPU pods（第6章讨论）可运行特定的机器学习工作负载和框架，磁盘和闪存设备可运行更专业的存储工作负载。

　　此外，工作负载根据其服务等级目标（SLO）被广泛分类为不同层级，这些目标对应服务可用性和延迟敏感性。例如，目前在Google我们使用以下工作负载层级：

1. HALO：高可用性延迟优化。该层级专为无法承受停机且不接受任何微小中断（例如共享核心导致的中断）的工作负载设计，因此提供最强的可用性SLO。典型示例包括GCE虚拟机、数据库或应用前端。

2. HALS：高可用性延迟敏感。该层级提供与HALO相同的可用性，但可容忍毫秒级的性能波动。示例包括数据仓库和流式数据管道。

3. HALT：高可用性延迟容忍。该层级提供与HALO相同的可用性，但针对可容忍数分钟延迟的应用。示例包括重要批处理作业或极少使用的容量储备。

4. MALS：主要可用性延迟敏感和MALT：主要可用性延迟容忍。这些工作负载是HA层级的成本优化变体，其容量可能偶尔被缩减以实现超配（参见第9章）。典型SLO可能是“1%的时间内您可能损失最多10%的资源”。

5. WALS：弱可用性延迟敏感和WALT：弱可用性延迟容忍。该层级提供更宽松的SLO，专为仅需吞吐量保证的长期批处理作业设计（“您将在一天内获得至少50%的请求任务小时数”）。

6. BE：尽力而为。该层级用于可无限期延迟的工作，该层级不预分配任何资源。若存在空闲资源，作业将运行，但可能被抢占且资源可能数日或数周不可用，因此无法保证特定作业最终会运行。

　　公共云目前除为虚拟机提供BE服务（如抢占式虚拟机或Spot虚拟机）外，尚未提供此类层级。通过层级管理容量可带来显著成本节约，但会增加显著复杂度，因此更适合云服务提供商而非云消费者。

### 3.1.3 公共云与超大规模

　　尽管最初设计用于Google搜索、Facebook或YouTube等在线数据密集型网络工作负载，仓库规模计算机（WSC）如今也支撑着Amazon、Google和Microsoft等公司的公共云。目前公共云已成为WSC中运行的最大工作负载，专门用于公共云的数据中心总容量远超私有超大规模云所使用的总容量。事实上，除Google和Meta的服务外，几乎所有主流消费者应用如今都运行在公共云上。

　　“超大规模”一词已流行用于指代大规模基础设施（“超大规模数据中心”）。虽然这两个术语相关，但并非完全相同。例如，纯数据中心提供商可能建造与WSC规模相当的大数据中心，但仅提供建筑设施，不包含可将该建筑内所有服务器转化为仓库规模计算机的软件栈。然而，“超大规模企业”通常用于描述构建WSC的公司——无论是专为其自身超大规模应用（Meta）而建，还是为提供云计算（AWS、Microsoft），或两者兼有（Google）。因此，超大规模企业是指同时运营大规模数据中心和基于这些数据中心构建的大规模服务的企业。

　　乍看之下，公共云计算似乎与超大规模应用截然不同：后者单个应用可能运行在数万台服务器上，而大多数运行在公共云上的企业应用仅使用少量服务器。然而，从WSC堆栈的许多部分来看，公共云如同一个消耗数万台服务器的大型应用。例如，在Google的基础设施中，GCE（GCP提供的虚拟机服务）看起来就是一个单一的大型应用。Google是唯一同时拥有超大规模消费者应用和大型公共云的企业，在本章中我们将说明同一基础设施如何同时支持大型内部在线服务和大型公共云产品。
