## 4.3 基础设施管理

　　一个仓库规模计算机（WSC）集群包含数百万个物理部件、数千万个计算核心以及至少数万个二进制文件。在如此规模下管理资源本身已成为一门艺术。如今，许多能够降低管理小型部署工作量的实践，最初都是为WSC开发的。

### 4.3.1 资源分配

　　使用公共云背后WSC的大多数开发人员，永远不需要处理其虚拟机、容器或存储所依赖的物理基础设施。分配新资源只需一次API调用，资源数秒后即可可用。在幕后，这种"始终可用容量"的幻觉与物理世界的现实相遇。我们如何确保客户请求时总有可用资源？

　　首先，我们需要规划系统来计算当前需求（今天应订购什么）以及预测未来需求（未来几个月到几年需要什么）。请注意，"当前"需求并非指即时需求，而是指需要立即订购的需求，即处于采购前置期的需求。例如，如果服务器从下单、运输到安装需要一周时间，那么我们必须提前知道一周后的需求量。

　　因此，前置期是资源分配的核心指标：在前置期内产生的需求通常无法满足，因此更短的前置期更有利。对于某些资源如单个服务器，在一定范围内实现数天的前置期是可能的（单台服务器的需求与1000台服务器的需求截然不同！）。而对于其他资源，供应链可能更长。例如，数据中心的建设通常需要约一年时间，某些组件如公用事业级变压器从下单到交付需超过一年。即使是小型组件如芯片也可能有长前置期：截至2024年，台积电5nm工艺制造的芯片需要约5个月！

　　缩短或隐藏前置期的一种方法是提前下单并保持缓冲库存。例如，通过保持一个季度的5nm芯片缓冲库存，非预期需求的响应速度可从5个月大幅缩短。但反面同样成立：我们现在必须提前至少三个月确定该芯片的需求结束，以便清空缓冲库存。即便预测完美，缓冲库存也可能成本高昂：每美元用于储备库存的资金，都是用于货架上静置物品的资金，这几乎不是有效的资金使用。因此，缓冲库存虽有必要，但并非完整解决方案。

　　规划过程的输出成为履约的输入。在供应链领域，履约被定义为接收、准备和交付订单的过程。对于WSC供应链，这可能包括订购组件（通常是"关键A类"部件，即最昂贵部件）并分配给制造商（制造商采购B/C类部件）、预订制造和运输产能、规划部署（下周二该地点是否有足够技术人员？）等。

　　在WSC中，供应可能不仅来自新采购。例如，某个大用户可能刚搬出同一（或附近）数据中心，其设备可用于满足新需求。或者，A地的服务可迁移至B地（B地有闲置资源），从而释放A地的资源以满足新需求。或者需求可重塑：一个请求100个A型GPU的任务可能同样能在150个B型GPU上运行，从而避免额外采购A型GPU。总体而言，资源池越大、需求在位置和具体硬件上的灵活性越高，履约总成本就越低。由于优质履约与普通履约的成本差异可能很大，规划和履约已演变为高度复杂的多维优化问题。或者，正如有人所说："规划和履约很简单，直到你亲自尝试。"

### 4.3.2 配置与可观测性

　　工作负载和硬件基础设施的规模与复杂性使得管理与可观测性框架成为任何WSC部署的基本组成部分，因此我们在此做更详细说明。

#### 4.3.2.1 配置管理

　　历史上，应用配置更多被视为次要事项而非专业领域。二进制文件通过命令行参数、shell环境变量和一些配置文件接收配置。现代软件开发改变了这一状况，使配置管理成为软件工程的重要组成部分[34]。

　　配置管理帮助工程团队通过自动管理配置数据的工具构建稳健稳定的系统。配置数据可能包括底层机器数据（例如特定类型机器的N核MGB内存）、应用参数、API密钥等凭证、应用所需服务的名称或端点等。

　　配置管理的目标是将所有这些数据存储在版本控制系统中，以实现变更审查、错误配置回滚和历史记录（"去年我们使用了哪些设置？"）。这类系统通常构建在源代码控制系统之上，因为它们的设计目的非常相似。这就是为什么这种方法常被称为"配置即代码"：将配置文件视为源代码文件进行管理。

　　“基础设施即代码”（IAC）进一步提出，服务实例的创建或删除应完全自动化。该方法不仅需要配置即代码，还需包含创建虚拟机或容器镜像、启动相应虚拟机或容器、配置网络与存储等所有步骤。IAC 工具示例包括 Ansible [35]、Chef [36]、Puppet [37] 和 Terraform [38]。

#### 4.3.2.2 可观测性

　　在监控（即收集和报告定义系统健康状态的指标）之外，可观测性要求系统能够解析监控数据以定位问题根本原因。监控收集单个组件的数据，而可观测性则从整体视角审视分布式系统。

　　系统运维人员必须跟踪互联网服务满足其服务等级指标（SLI）的程度。SLI 衡量特定服务方面，例如某 API 调用 C 成功执行的时间百分比。服务等级目标（SLO）则规定服务应达到的目标 SLI（“C 的成功执行率应达到 99.9%”）。最终，服务等级协议（SLA）是法律协议，明确 SLO 以及未达成 SLO 时的违约责任（“若我们未能满足此 SLO，您可无责终止合同”）。

　　监控信息必须非常及时，以便运维人员（或自动化系统）能快速采取纠正措施，避免重大中断——理想情况下应在数秒内而非数分钟内。有时最关键的信息仅限于前端服务器收集的少数信号，例如用户请求的延迟和吞吐量统计。最简单的监控系统可以是脚本，每隔几秒轮询所有前端服务器获取相应信号，并在仪表盘上展示给运维人员。Google Cloud Operations 工具 [39] 提供类似产品用于监控、排查和运维云服务。负载均衡器常用于自动解决某些问题，例如将流量重定向至健康服务器池，或向池中添加额外服务器。

　　大规模服务通常需要更复杂且可扩展的监控支持，因为前端数量可能庞大，且需要更多信号来表征服务健康状态。例如，可能需要收集信号本身及其随时间变化的导数。系统还可能需要监控除延迟和吞吐量外的业务特定参数（“是否有购买按钮点击失败？”）。监控系统通常支持表达式语言，允许运维人员基于基线信号创建派生参数。最终，系统会根据监控值和阈值向值班运维人员生成自动警报。精细调整警报系统颇具挑战性，因为频繁误报会导致运维人员忽视真实警报，而仅在极端情况下触发的警报可能延误问题解决。《SRE 书籍》[40] 对监控与警报进行了深入讨论。

　　WSC 开发者还需应用日志和崩溃日志以确定问题根源。传统（单机）应用中这些日志存储于本地文件系统，但在 WSC 中需集中收集到中央仓库，以便开发者跨服务所有实例查看应用日志。通常，此类文件先写入本地存储，日志收集进程将其复制到中央仓库并保留特定时长。该过程通常还会压缩日志并“净化”敏感数据，例如可能存在于崩溃转储中的加密密钥。

#### 4.3.2.3 追踪工具

　　尽管服务级仪表盘帮助运维人员快速识别服务级问题，但通常缺乏了解服务为何缓慢或未满足需求的详细信息。运维人员和服务设计者都需要工具来理解众多程序（可能运行在数百台服务器上）之间的复杂交互，从而确定性能异常的根本原因并识别瓶颈。

　　分布式系统追踪工具满足这一需求，是实现从监控到可观测性的重要步骤。这些工具旨在确定分布式系统为特定发起者（如用户请求）执行的所有工作，并详细描述各组件间的因果或时间关系。例如，当用户点击“购买”按钮时，应用前端可能在完成购买过程中调用数百个微服务。因此，若流程失败，为何失败？追踪工具得名于其功能：它们旨在追踪请求通过所有子请求直至完成的全过程，生成所有参与服务的完整记录及其性能表现。

　　追踪工具可分为两大类：黑盒监控系统和应用程序/中间件插桩系统。WAP5 [41] 和 Sherlock 系统 [42] 是黑盒监控工具的典型代表。其工作原理是通过观察系统组件间的网络流量，并利用统计推断方法推导因果关系。由于将所有系统组件（除网络接口外）视为黑盒，这些方法无需了解或依赖应用程序或软件基础设施组件即可运行。然而，这种方案本质上会牺牲信息准确性，因为所有关系都必须通过统计推断得出。收集和分析更多消息数据可以提高准确性，但会增加监控开销。

　　基于插桩的追踪方案在 Pip [43]、Magpie [44] 和 X-Trace [45] 等学术系统中率先提出，其优势在于能够通过显式修改应用程序或中间件库，在跨机器及机器内部模块边界传递追踪信息。这些标注模块通常还会将追踪信息记录到本地磁盘，供外部性能分析程序后续收集。此类系统具有很高的准确性（无需推断），但要求分布式系统的所有组件都进行插桩以收集完整数据。Dapper 系统 [13] 是一种基于标注的追踪工具，通过插桩与所有应用程序通用的关键模块（如消息、控制流和线程库）实现对应用层软件的有效透明性。GCP 的 Cloud Trace [46] 是 Dapper 系统的公开实现，接受任何 OpenTelemetry 兼容源的追踪数据。

　　迄今为止我们讨论了 RPC 层面的追踪，但该概念同样适用于函数调用。例如，LLVM 编译器的 XRay 功能通过编译器插桩在每个函数入口和出口添加追踪点，启用时可提供极细粒度的延迟细节，禁用时则具有极低开销。其他工具可提供更深入的内省功能，例如允许用户在不重新编译或部署的情况下动态添加日志记录。

#### 4.3.2.4 性能工具

　　追踪工具对性能优化也极具价值。由于追踪数据包含时间戳信息，我们可以深入了解影响关键服务（如购买按钮点击）延迟的因素。此外，当追踪数据持续收集时，它们有助于理解服务在不同负载下的性能表现，或解释为何区域 A 的服务实例性能与区域 B 存在差异。

　　基于硬件性能计数器采样的 CPU 分析器在帮助程序员理解微架构性能现象方面取得了巨大成功。由于 WSC 应用运行在多台机器上，此类工具需要对多台机器进行分析。Google-Wide Profiling (GWP) [47] 会随机选择部分机器收集短时整机及进程级分析数据，并结合所有 Google 二进制文件的符号信息仓库，生成集群级分析视图。GWP 可回答诸如"Google 上最频繁执行的程序是什么？"或"哪些程序是内存使用大户？"等问题。Google Cloud Operations 工具产品 [39] 即受 GWP 启发。

　　性能工具不仅用于观测，还可主动优化。例如，基于分析的优化 [48] 会利用生产环境的分析数据优化同一二进制文件的未来版本。优化的内存分配器可减少缓存和 TLB 命中缺失 [49]。该领域进展尤为困难，因为优化的真实性能影响只有在大规模部署时才显现，且可能与小规模实验环境中的收益大相径庭 [50]。例如，过度内联或特化分配器可能在小型基准测试中显著提速，但在生产环境中却可能因数百万个内联分配器导致的代码膨胀引发过多指令缓存缺失，反而降低性能。

### 4.3.3 实践：DevOps 与 SRE

　　WSC 及后续的公共云不仅要求应用的设计和实现采用新方法，也要求运行方式的革新。当前最流行的两种新方法是 DevOps（开发运维一体化）和 SRE（站点可靠性工程）。

　　DevOps 将开发与运维整合为单一团队：开发者不再将成品移交给运维团队运行维护，开发工程师与运维工程师之间也不再有明确界限。DevOps 的目标是通过让同一团队负责开发与运维，缩短从代码提交到生产部署的时间，使其能够平衡双方需求。例如，若缺乏足够的应用级指标或日志，调试将变得困难；而由于同一工程师需处理生产环境的问题或故障，他们更倾向于构建高可观测性的应用。

　　反对 DevOps 的观点认为两种职业所需技能差异过大：优秀的软件工程师未必擅长可靠性、运维严谨性或容量规划。因此 SRE 采用不同方式解决相同问题：仍存在两个团队，但"运维"团队由专注于可靠性及其他运维属性（如可扩展性、效率、安全性等）的软件工程师组成。

　　SRE书籍第1章[40]对SRE原则进行了优秀概述，这些原则可总结为一句简单的话：SRE是当您让软件工程师设计运维团队时所发生的结果。SRE始终专注于工程实践（例如自动化解决方案或解决根本原因，而非仅处理症状），在实现服务SLO的同时追求最大变更速度。换句话说，目标并非达到100%正常运行时间，因为这可能导致SRE抵制变更——而变更恰恰是导致停机的首要因素。相反，SRE哲学要求"合理使用停机预算"（并保持在预算范围内，即达成SLO）。如果发生停机，应由新出现的问题引起，而非"简单错误"。例如，若停机由配置文件损坏导致，这表明服务未能正确验证其配置。

　　在仓库规模计算机（WSC）层面运营还需要将故障视为学习机会的强文化。在百万级核心和数千名软件工程师的系统中，总会出现问题。即使拥有最佳团队、最佳技术和最佳实践也是如此。因此，目标是构建一个即使硬件、软件和人员存在缺陷也能高度可靠的系统。实现高可靠性的关键机制之一是无责复盘[40]。无责复盘是SRE文化的基石。真正的无责复盘必须聚焦于识别事件的促成因素，而非追究任何个人或团队的不当行为。无责撰写复盘的前提是假设所有涉事人员在现有信息基础上都做出了合理判断。如果盛行指责和羞辱个人或团队的"错误"行为的文化，人们将因惧怕惩罚而隐瞒问题（详见[40]第15章）。虽然本书主要涉及硬件和软件主题，但建立和维护正确的工程文化往往更具挑战性且更为重要！我们将在第11章回顾二十五年WSC发展时对此进行更深入探讨。
