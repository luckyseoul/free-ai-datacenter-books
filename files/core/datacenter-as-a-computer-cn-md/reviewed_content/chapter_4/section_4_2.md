## 4.2 分布式系统

　　在单一系统中，操作系统层管理资源并向该单一计算机提供基础服务。类似地，由数千台计算机、网络和存储组成的WSC系统也需要软件提供更大规模的类比功能。本节讨论该软件最重要的组件：集群级系统、存储以及广域网和边缘网络。

### 4.2.1 集群级基础设施

　　许多WSC通过全球网络互联并提供全球服务。集群代表作为管理单元的最小域。一个集群（有时也称为单元）由连接到单个高带宽集群网络（如Jupiter，第6章）的机器组成。通常，集群包含位于单一数据中心大厅或建筑中的数万台服务器（即10-30兆瓦功率）。集群中的所有服务器通过相同网络和单一控制平面连接，因此存在同时不可用的可能性。

　　集群的层级低于区域（参见第2章），因为一个区域可以包含跨多个建筑分布的多个集群。例如，新区域可从单个集群开始，随着时间推移逐步添加更多集群。

#### 4.2.1.1 资源管理

　　资源管理可能是集群级基础设施层中最不可或缺的组件。它控制用户和作业到硬件资源的映射，执行优先级和配额策略，并提供基础任务管理服务。资源管理具有四项明确职责：

- **硬件管理** 允许添加、升级、维修和移除硬件。有效管理硬件资源需要库存管理（例如，哪些设备在机房、位置及内部组件）、状态管理（例如，硬件是否通电且健康）以及工作流工具（例如，指引技术人员定位服务器、标识待更换组件并记录新旧组件的条形码）。
- **机器状态管理** 处理存储在每个设备本地的数据位，例如固件镜像、硬件配置参数（如时钟频率设置）和设备身份（如证书）。
- **作业管理** 将工作负载调度到单个机器上，并响应供应变化（如机器不可用）或需求变化（如作业需要扩展以处理额外负载）。根据向用户提供的抽象层级，调度器可能分配裸金属服务器（原始机器）、虚拟机（通常占用部分机器资源）或容器（进一步抽象虚拟机）。
- **容量管理** 将资源管理扩展到物理供应链，平衡资源过剩（为闲置资源付费）与资源不足（因关键资源短缺影响应用）的成本。与作业管理的亚秒级分配决策不同，容量管理的时间范围从数天到数月不等，取决于硬件资源的前置时间。

　　在最简单形式中（仍用于小规模企业），资源被手动静态分配给特定用户或作业，通常导致利用率低下。相比之下，WSC资源管理必须高度自动化，既出于必要性（手动跟踪数百万资源不可行），也出于机会性（1%的效率提升可能代表数百万美元的节省）。

　　Google的Borg调度器[1, 2]将作业调度到异构服务器池中。每个作业代表一个服务（如搜索后端），包含多个任务。每个任务代表运行在单个机器上的单个二进制文件（附带配置文件）。因此，调度搜索后端作业可能涉及调度数千个任务，每个任务处理搜索索引的一个分片。所有机器共享使用，任务调度转化为装箱问题，目标是将所有任务（来自所有作业）调度到机器池中，使内部碎片（即闲置资源）最少。例如，若任务需要32核和40GB内存，将其调度到剩余32核和100GB内存的服务器上会浪费剩余60GB内存。

　　作业可指定调度约束。亲和性（或反亲和性）约束规定特定任务应（不应）共调度到同一台机器、交换机或电源域；例如，具有两个副本的数据库作业希望其从副本位于与主副本不同的机器和交换机上，以避免单点故障影响两个副本。其他约束可能包括机器类型，以优化效率和性能。部分约束可能基于性能分析信息自动生成。例如，Borg生态系统包含识别作业在特定硬件上运行良好（或差）的工具，并据此调整调度器偏好。

　　Kubernetes[3]是一个流行的开源容器管理器，其概念源自Borg（事实上，初始Kubernetes团队主要由曾参与Borg的工程师组成）。Kubernetes旨在高度可移植和可扩展，可在多种公有云、私有云以及物理和虚拟机上运行。与Borg不同，它不涵盖机器本身的生命周期，因此假设底层基础设施提供虚拟机。

　　尽管许多 Kubernetes 用户将其视为云 API 抽象层，但一些用户希望实现更高级别的抽象。Kubernetes 包含众多扩展点，包括 API 内部的扩展点，这使得组织能够为其内部用户创建定制化平台。这些内部开发者平台允许企业嵌入并强制执行自身关于服务运行方式的观点。Kubernetes 的可扩展性也催生了多个开源 PaaS 和函数即服务（FaaS）项目，并影响了其他容器化产品。例如，GCP 的 Cloud Run [4] 提供了一种类似 Kubernetes 的抽象层，不仅隐藏了虚拟机，还隐藏了大部分操作细节，仅需用户提供容器镜像即可将其作为服务运行。

　　如今，资源管理器也集成了电力、能源和碳排放优化功能。例如，它们允许对电力基础设施进行功率超配（见第9章讨论），在需要时限制低优先级工作负载 [5]，跨区域调度器可能会将工作负载迁移到碳排放最低的区域 [6]。

#### 4.2.1.2 基础设施软件

　　几乎所有大规模分布式应用都需要一组基础功能。例如可靠分布式存储、远程过程调用（RPC）、集群级同步与锁定。在大型集群中，正确实现这类功能并保证高性能和高可用性非常复杂。明智的做法是避免为每个应用重新实现这些复杂代码，而是创建可复用的模块或服务。Colossus（GFS 的继任者）[7, 8]、Dynamo [9] 和 Chubby [10] 是大型集群中可靠存储和锁定服务的典型例子。

　　许多在小型部署中简单的手动操作任务，在大规模系统中需要大量基础设施支持：软件镜像分发与配置管理、监控、紧急情况下的告警排查等。Autopilot [11] 为 Windows Live 数据中心提供了一些此类功能的设计示例。监控硬件集群的整体健康状态也需要精细的监控、自动诊断和修复流程自动化。Google 的 System Health Infrastructure [12] 是高效健康管理所需的软件基础设施示例。最后，此类规模系统的性能调试与优化也需要专用解决方案。Dapper [13] 是首个用于收集 RPC 跟踪以调试性能问题的实际系统，并启发了当今的 Open Telemetry 生态系统 [14]。

### 4.2.2 非结构化存储系统

　　Google 的 GFS [7] 是首个具有简单文件类抽象的集群级存储系统。最初，GFS 被设计用于支持单一用例——网络搜索索引系统（将爬取的网页转换为搜索索引文件的系统）。因此，它专注于为数千个并发读写器提供高吞吐量，并在高磁盘故障率下保持稳健性能。GFS 用户通常处理大量数据，因此 GFS 进一步优化了大块读写操作。

　　系统架构包含一个处理元数据操作的主服务器，以及运行在每个带磁盘的服务器上的数千个 chunkserver（次级）进程，用于管理这些磁盘上的数据块。在 GFS 中，容错通过跨机器复制实现，而非传统 RAID 系统中的单机内复制。跨机器复制使系统能够容忍机器和网络故障，并实现快速恢复，因为特定磁盘或机器的副本会分布在数千台其他机器上。

　　GFS 通过多项设计加速部署。最显著的是，文件系统的元数据必须能完全存放在单台服务器内存中。这一限制大大减少了实现工作量，但也限制了存储单元的规模和中央元数据服务器故障时的正常运行时间。随着 GFS 需求的扩展（如 Gmail 和 YouTube 用例），GFS 被其继任者 Colossus [15] 取代。

　　Colossus 单元由多个文件夹组成；每个文件夹本质上是一个独立的文件系统。每个文件夹的元数据以独立的 BTree 层级结构存储，该层级称为 Metadata1 或 MD1。MD1 BTree 文件的元数据存储在另一个名为 MD0 的 BTree 层级中，而 MD0 树的元数据（即 MD0 文件的文件描述符）则存储在基于 Paxos [16] 实现的分布式日志系统中。（为简化起见，图4.2省略了 MD0 和 MD1。）

图4.2 Colossus 文件系统的简化架构

　　Curators 实现了一个高可用、水平可扩展的元数据服务器。它们负责管理命名空间、文件系统结构及相关元数据（访问控制列表、时间戳等）。Curators 处理文件操作（创建、删除、重命名等），但不参与读写操作。Paxos 决定哪个 Curator 是该文件夹的领导者（能够修改元数据的服务器）。

　　所有数据存储在名为 D 的底层系统中，该系统实现了一个类似于网络附加磁盘的简单 API，因此得名。D 并非分布式文件系统；每个 D 服务器仅提供对其本地存储设备上文件的网络访问。除了每个文件的校验和外，D 不提供任何实质性的可靠性保证。任何形式的复制、纠错或其他备份系统都必须由使用它的系统（即 Colossus）实现。

　　Colossus客户端库实现了整体功能的重要部分，包括复制、负载均衡以及处理无响应的D服务器。例如，如果一个文件采用三副本复制（r=3），客户端负责将写操作发送到三个副本。将这些功能置于客户端中可实现有效的可扩展性：文件系统的读写客户端越多，可用于读写操作的CPU和网络带宽就越多，整体系统性能也不太可能因实际Colossus服务器而成为瓶颈，因为它们仅参与部分操作，如打开文件。

　　GFS的初始版本仅支持简单的复制，但Colossus增加了对空间效率更高的里德-所罗门码（Reed-Solomon codes）的支持，这种编码方式在相同可用性水平下，相比简单复制可将复制的空间开销降低约两倍（参见第3.3.2节）。此外，Colossus将文件块分布在整个集群中，使得相关联的故障极不可能导致数据丢失（例如，副本存储在不同的机架和不同的网络区块中）。

　　在整个集群中广泛分布块也能加速恢复过程。由于存储在特定设备上的块副本被分散到存储集群中的许多机器上，丢失块的重建可以在高并发下快速完成。快速恢复非常重要，因为较长的恢复窗口会使副本不足的数据块面临数据丢失风险，特别是在集群发生额外故障时。关于Google分布式文件系统可用性的全面研究可参考Ford等人[17]。Google文件系统设计演进的详细讨论也可参考McKusick和Quinlan[15]。

　　将大量功能集成到Colossus客户端中存在一个显著缺点：要修复错误或添加新功能，所有Colossus客户端都需要重新编译并重启！这显然是一个非常不理想的特性，因为通常无法控制使用文件系统的应用程序的升级节奏。然而，对于单一公司（即云服务提供商Google）内部使用而言，这是一个实用的选择：在Google，标准策略要求出于多种原因定期重新构建所有二进制文件，该策略同样确保了Colossus客户端的更新。不过，Colossus的公共版本GCS[18]做出了相反选择，因为无法强制云客户及时更新其应用程序。因此，GCS客户端库较为简单，仅将请求转发至包含实际Colossus客户端的API服务器。

### 4.2.3 结构化WSC存储

　　Colossus和GCS的简单文件抽象可能足以满足处理大数据块的系统需求，但应用程序开发者还需要数据库类功能，即数据集能够被结构化和索引，以便进行小规模更新或复杂查询。结构化分布式存储系统（如Google的Bigtable[19]和Amazon的DynamoDB[9]）正是为满足这些需求而设计的。与传统数据库系统相比，Bigtable和DynamoDB牺牲了一些特性（如模式表示的丰富性和强一致性），以换取在大规模场景下的更高性能和可用性。例如，Bigtable提供了一个简单的多维排序映射，由行键（字符串）关联多个按列组织的值，形成分布式稀疏表空间。列值通过时间戳关联以支持版本控制和时间序列。即使与性能最高的传统SQL数据库相比，Bigtable的吞吐量也可达到多个数量级的提升（每秒数十亿次查询）。

　　然而，Bigtable和DynamoDB仅提供最终一致性，将实现完全一致性的负担转移给了使用这些系统的应用程序。特别是多步骤操作可能变得难以编写且容易出错。例如，在最终一致性的系统中，应用程序可能写入新数据，但随后在后续读取时看到旧数据，如果第二次读取访问的是与初始写入不同的副本。第二代结构化存储系统如Megastore[20]和尤其是Spanner[21]正是为解决此类问题而设计的。Megastore和Spanner在提供更丰富的模式和类SQL功能的同时，也提供了更简单、更强的一致性模型。Megastore通过同步复制牺牲了写入吞吐量，限制了其可用性和采用率。

　　Spanner使用一种名为TrueTime的新时间API，能够在不牺牲水平扩展能力的前提下高效序列化全球分布式事务，为需要无缝广域复制容错的应用程序提供简单的一致性模型。TrueTime是一个基于GPS和原子钟的全球分布式时钟，能够提供严格的时间不确定性边界。与返回单一时间点不同，TrueTime为每台机器提供一个时间区间[earliest, latest]，其中包含真实时间。Spanner利用TrueTime为事务分配时间戳。通过依赖这些全局一致的时间戳，Spanner可以确定事务的顺序，即使这些事务源自全球不同服务器。

　　Spanner 提供外部一致性，这是事务处理系统中最严格的致性属性。Spanner 中的所有事务都满足这一致性属性，而不仅仅是分区内的事务。外部一致性表明，Spanner 执行事务的方式与事务串行执行的系统无法区分。此外，如果一个事务在另一个事务开始提交之前完成，系统保证客户端永远不会看到包含第二个事务效果但不包含第一个事务效果的状态 [22]。

　　Spanner 提供完全一致的类 SQL 数据库，具备水平扩展能力和极高的可用性（99.999% SLO）。这似乎违反了 CAP 定理 [23] 所施加的限制，该定理指出系统只能实现以下三个属性中的两个：一致性（Consistency）、可用性（Availability）和网络分区容忍性（resilience to network Partitions）。NoSQL 系统必须牺牲一致性以获得其他两个属性，那么 Spanner 如何同时提供这三个属性？实际上，Spanner 仅提供 CP（放弃可用性），但在实践中其可用性如此之高，以至于用户可以将其视为 CA 系统 [24]。这种组合在实践中是可能的，前提是能够完全控制网络——这在广域网（WAN）中很少见，但在 Google 的全球网络中得以实现。即便如此，它仍需要显著的网络路径冗余、架构规划以管理相关故障，以及非常谨慎的操作，尤其是在升级过程中。如果发生中断，Spanner 会优先选择一致性而非可用性。

　　由于在一致性和可靠性方面的优势，Spanner 已成为几乎所有 WSC 用户的事务存储系统首选。例如，在 Google，Gmail 和 Google Photos 将所有数据存储在 Spanner 中，Google 的点击流和广告数据管理系统也是如此。对于后者，Google 的 F1 系统 [25] 使用 Spanner 作为数据存储，通过分布式 SQL 查询、事务一致性二级索引、异步模式变更、乐观事务以及自动变更历史记录和发布等数据库特性管理所有 AdWords 数据。

　　在结构化存储光谱的另一端是几乎完全专注于高性能的系统。此类系统通常缺乏对事务或地理复制的支持，采用简单的键值数据模型，且可能具有较弱的持久性保证。Memcached [26] 作为分布式 DRAM 基础的对象缓存层开发，被广泛用于简单缓存。一旦其价值变得明确，快速键值存储在学术界引起了广泛关注。例如，斯坦福大学的 RAMCloud [27] 系统使用分布式 DRAM 基础的数据存储，但目标是实现更高的每节点性能和存储节点故障下的持久性。FAWN-KV [28] 系统也提出了高性能键值存储系统，但改用 NAND 闪存作为存储介质，并额外强调能效，这一主题我们在第 9 章中将更详细地探讨。如今，多个开源键值存储系统广受欢迎，包括 Redis ([29])。

### 4.2.4 广域网与边缘网络

　　尽管本书主要讨论 WSC 作为大型构建模块，但大多数 WSC 都是超大规模服务商或公共云提供商全球基础设施的一部分。此处我们将概述 WSC 本身之外的基础设施；虽然规模较小，但它对 WSC 价值的影响却不成比例。

　　我们以德国用户访问 www.google.com 的请求为例来说明这一基础设施。第一步，Google 的 DNS 服务器需要为该端点返回一个 IP 地址，最好位于用户附近。如今，这通过 Anycast DNS [30] 实现：同一 IP 地址（例如 1.2.3.4）从所有拥有服务器的区域广播，互联网的标准边界网关协议（BGP）路由将用户请求导向网络跳数最少的区域（例如汉堡）。Google 既将 Anycast 用于 DNS 服务器本身，也用于接收请求的前端 IP。

　　另一种方法是使用智能 DNS 服务器，根据请求的源 IP 计算最佳位置。这种方法依赖于全局网络数据库，将 IP 范围映射到地理位置；例如，它可能知道用户的 IP 属于德国电信在汉堡使用的子网，因此将请求导向汉堡区域。Google 的 DNS 在切换到 Anycast 之前曾采用此方法。基于 IP 的名称解析的主要缺点在于构建和维护网络地图的复杂性，以及在 DNS 请求时信息不足的固有局限性。特别是，到达 Google DNS 的 DNS 请求往往并非直接由用户发送，而是由用户 ISP 或公司拥有的中间（递归）DNS 解析器发送，这些解析器可能距离用户很远。因此，DNS 请求的源 IP 地址可能无法准确反映用户的真实位置。（HTTP/3 规范现在包含了一个选项，允许服务器指定不同于当前 IP 的首选 IP 地址 [31]，但截至 2024 年，该功能尚未广泛实施。）

　　用户的请求随后到达边缘位置，所谓“边缘”是指网络的边界（外缘），而非核心区域（例如实际的 WSC）。"Edge" 是一个模糊术语，其含义因上下文而异。例如，有时指客户端设备（即边缘计算在用户的手机或笔记本电脑上进行），有时指高度分布式的网络位置（如社区网络机柜或基站），或指大城市中的网络 POP（网络接入点）。

　　Google的网络具有两层边缘服务。Google全球缓存（GGC）[32]通过边缘缓存、内容压缩和智能路由技术，快速高效地传输内容。大多数GGC节点仅由几台服务器组成，并部署在合作伙伴ISP网络内部。从这个意义上说，它们位于Google网络边缘之外，因为它们既不直接连接Google网络，也不托管在Google场地内。

　　GGC是内容分发网络（CDN）的一个典型示例，这类网络专门设计用于分发静态内容（图片、视频）或直播内容。GGC通过缓存机制可显著加速内容传输（如YouTube视频或Spotify歌曲），同时降低ISP的基础设施成本。全球范围内存在数万个GGC节点；图4.3展示了Cloud Media CDN中规模较大的缓存节点分布。

　　第二层边缘服务位于真正的网络边缘，通常部署在互联网交换点或网络POP[33]；全球此类节点超过百个。继续以德国为例，截至本文撰写时（2024年），Google在德国运营五个边缘节点，包括汉堡节点，因此用户的请求很可能由汉堡边缘POP处理。边缘POP的规模因服务区域而异，通常包含处理或路由请求的前端服务器，以及为热门内容提供二级缓存的缓存服务器。无法本地处理的请求将转发至最近的可用服务器数据中心。例如，GCP请求可能转发至法兰克福或柏林GCP区域，但搜索请求可能转发至邻近的搜索服务数据中心（如丹麦或荷兰）。

　　除了处理请求外，网络POP还部署对等路由器，用于连接Google网络与该位置的ISP网络。"对等连接"指两个网络之间达成的直接互联协议，双方无需支付费用。对等连接可提升双方网络性能，因为它能提供流量水平的直接可见性；例如，若汉堡的ISP A端口过载，Google可将流量重定向至其他较不拥堵的端口。

　　不同网络具有不同的路由和对等策略。"热土豆"路由通过尽早将数据包送出网络来最小化网络成本；例如，若数据包目的地为ISP B但本地ISP A可路由至B，则数据包立即交由A处理，随后通过公共互联网（可能经过多个ISP）传输至B。"冷土豆"路由则相反，将数据包保留在内部骨干网直至接近用户位置时才交由B。通过冷土豆路由策略、开放对等策略及在多数对等位置的部署，Google超过98%的数据包仅需一次从Google网络直接交付至终端用户ISP或企业网络的切换。

　　边缘POP通过网络骨干网相互连接，并与主数据中心相连（图4.4）。该骨干网采用分层架构；例如，大都会区域的多个POP通过城域网互联，该网络又在一处或多处连接区域骨干网，区域骨干网进而连接全球骨干网（图4.5）。在许多骨干网中，数据中心间的流量占比远高于数据中心与用户间的流量，这是由于区域间的数据复制需求。

　　如第2章所述，区域包含多个校区内的多个可用区，因此网络骨干网连接的是区域而非数据中心。区域骨干网连接区域内所有校区；通常这些校区间距仅数十公里，因此区域骨干网可低成本提供高带宽。包含多个建筑的大型校区还部署校区网络，通过更高带宽、更低延迟的连接将建筑互联。这些网络层级（如校区和区域）通过网关互联，网关负责路由跨层级流量。虽然功能相似，但网关规模差异显著：单个对等路由器组成的网关可连接多个ISP至网络，而多拍比特校区网络与多拍比特区域网络之间的网关则包含数十个机箱、数千个端口，更接近集群网络而非互联网POP。

　　回顾示例，用户发送至Google的数据包将通过其ISP网络传输至Google边缘位置的对等路由器，该路由器连接ISP与Google网络。若无法本地处理，数据包将经Google城域网传输至骨干网网关，再传输至数据中心附近的另一个骨干网网关。随后继续通过目标城域网传输至校区网络网关，最终到达托管服务的建筑，在那里另一个网关将其路由至实际服务器所在的集群网络。
